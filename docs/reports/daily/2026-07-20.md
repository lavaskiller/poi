## 1. Problem framing (end of day)

POI identification is **two stages**:

1. **Retrieval / coverage** — MapKit nearby supplies candidates (GPS).
2. **Selection** — pick which candidate the **photo** was taken at.

Today’s product rule for selection (PWE-13):

> Do not primarily rank by distance. For each candidate, ask whether the photo
> could have been taken **at that place**; pick the one that most clearly fits.

Distance remains how the list is built; it is only a tie-break / fallback after
visual evidence fails.

## 5. Engineering process (method used today)

```text
1. Lock data
   - active MapKit snapshot + reconciled CSV
   - re-measure nearest (honest floor)

2. Separate failures (already in poi-case-types.html)
   - top5 present (selector) vs absent (retrieval/policy)
   - A empty / B container / C alias / D off-radius

3. Cheap selector loop (minutes)
   - nearest, weighted, OCR, access demote
   - offline bakeoff → official run_algorithm artifact

4. Reframe selector objective
   - photo–place match, not distance

5. Expensive vision only on residual misses
   - FastVLM place_match on 33, cascade stitch
   - no regression on already-correct cheap path

6. Human visual triage on residual 27
   - solvable vs hard vs label policy
```

Principles:

- **Never mix retrieval failure into selector accuracy.**
- **Cascade:** cheap exact wins first; VLM only where needed.
- **Strict name** metric stays; alias/related reported separately.
- **Rename by role** (`selector_*`, `vlm_topk_*`), version in metadata/run name.
