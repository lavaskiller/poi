## MVP scoring decisions

- Korea is excluded for now because Kakao Local canonical/candidate data is not populated yet.
- Non-Korea rows are evaluated against MapKit.
- MVP matching uses exact string equality within the same provider.
- `provider_place_id` is nullable/optional and is not required for MVP scoring.
- Normalized matching is retained only as a fallback/evidence view; it does not replace the primary provider-aware exact-name policy.

## Why the GitHub repo looked messy

The repository was split out of an active prototype workspace. The first priority was to make it safe to publish by removing private/raw/generated artifacts, then to wire real MVP metrics. It was not yet reorganized into a clean package layout.

Main reasons:

- `mvp-eval-ui.html` is a prototype single-file UI with CSS/HTML/JS together.
- `server.py` is a local dashboard server and still depends on local data files that are intentionally not public.
- Some older docs were written before `/api/matchrate` and `match_score.py` existed, so status text had drifted.
- Public repo intentionally excludes the raw data needed to reproduce private metrics.

Cleanup chosen for now:

- Do not rewrite public Git history without explicit approval.
- Keep existing pushed commits.
- Add small forward commits for policy, docs, and reports.
- Defer structural UI/server refactoring until MVP behavior stabilizes.
