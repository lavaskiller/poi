## 4. MVP scoring policy

The scoring policy was made explicit and consistent across the evaluator, API, README, and UI:

- **Korea is held out** until Kakao Local canonical/candidate data is populated
  (status flag: `excluded_korea_pending_kakao`).
- **Non-Korea rows** are evaluated against **MapKit**.
- MVP matching uses **exact string equality within the same provider** (canonical GT
  name vs. candidate name).
- `provider_place_id` is nullable/optional and is **not** required for MVP scoring.
- Normalized/string-relaxed matching is retained **only as a fallback/evidence view** —
  it does not replace the primary provider-aware exact-name policy.
- The numbers are **candidate coverage/rank metrics**, not visual or user-intent
  identification accuracy.

## 5. What was implemented

### `tools/match_score.py` — provider-aware metrics engine
- Stdlib-only, no external dependencies.
- CLI: `python3 tools/match_score.py --json`.
- Dataset filtering (`all`, `linkedspaces`, `union-city`, `vancouver`).
- Exact/raw mode and normalized diagnostic mode.
- Legacy MapKit `app_poi_rank` support sourced from the current private reconciled CSV.
- Optional provider candidate JSONL loading.
- Converts MapKit TSV preview data (`ls_nearby_results.tsv`) into generated JSONL.
- Emits the Korea holdout status flag.

### `server.py` — live match-rate endpoint
- `GET /api/matchrate?dataset=all&mode=exact`
- `GET /api/matchrate?dataset=linkedspaces&mode=exact`
- `GET /api/matchrate?dataset=all&mode=normalized`
- The endpoint calls `match_score.evaluate()` directly rather than returning mock metrics.
- Configured to read private data paths for local match-rate computation.

### `mvp-eval-ui.html` + `mvp-eval-ui.js` — UI cleanup
- Evaluation cards now fetch `/api/matchrate` instead of static mock values.
- **Removed fake/mock algorithm rows** (`ocr-match`, `nearest-only`, `fastvlm-name`,
  and the hard-coded 22% / 15% scores).
- The run button no longer fabricates pretend executions; it states plainly that the
  submission harness is not implemented yet.
- The selection-accuracy chart stays empty until real algorithm predictions exist.
- JavaScript was moved out of the HTML into same-origin `mvp-eval-ui.js` so the strict
  `script-src self` CSP no longer blocks the UI.
- Static overview counts/bars were replaced with `/api/overview`-driven values.
- Case labels were renamed from identification-failure wording to candidate-rank
  wording (`rank1`, `rank>1`, `검색실패`) so MapKit is not misrepresented as
  visual/user-intent POI identification accuracy.
- UI labels clarify that current scoring excludes Korea and uses MapKit for non-Korea rows.

### `tools/validate_upload_package.py` — upload validation
- Validates user-filled dataset ZIP packages before ingest.
- The empty template ZIP is *expected* to fail validation because it has no manifest
  data rows; users must fill `photos/` and `manifest.csv` before uploading.

### Generated local artifact (not published)
- `generated/mapkit_candidates.jsonl` — 537 rows converted from `ls_nearby_results.tsv`.

## 7. Why the repo still looks messy

The repo was split out of an active prototype, not built greenfield. The first priority
was making it *safe to publish* (remove private/raw/generated artifacts), then wiring
*real* MVP metrics. Structural cleanup is intentionally incremental.

Known sources of messiness:

- `mvp-eval-ui.html` is still a single-file prototype (HTML + CSS + JS together, though
  JS is now split into `mvp-eval-ui.js`).
- `server.py` is a local dashboard server that still assumes local data files exist
  outside the public repo.
- Some older specs/reviews/plans were written before `/api/matchrate` and
  `match_score.py` existed, so their status text had drifted.
- Generated/private data is excluded, so some commands only work in the private workspace.

Cleanup decisions:

- Do **not** rewrite public Git history without explicit approval; keep pushed commits.
- Add small forward commits for policy, docs, and reports.
- Add dated work logs under `report/`.
- Defer structural UI/server refactoring until MVP behavior stabilizes.
