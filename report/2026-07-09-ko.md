## 1. 요약

오늘은 비공개 `poi-test-data` 프로토타입 작업공간에서 공유 가능한 POI 평가 도구를
분리해 깔끔하게 배포 가능한 **공개 저장소**로 만들고, MVP 매치레이트 경로를
**평가기(evaluator) → API → UI** 로 끝까지 연결했습니다.

핵심 성과 세 가지:

1. **안전한 공개 분리** — 코드·스펙·템플릿·보고서만 담고, 원본/비공개/생성 산출물은
   모두 제외한 새 공개 저장소를 생성.
2. **실제 지표 파이프라인** — 표준 라이브러리만 쓰는 provider-aware 평가기가 라이브
   `/api/matchrate` 엔드포인트에 연결되고, UI가 이를 직접 읽도록 하여 하드코딩된
   목(mock) 수치를 제거.
3. **정직한 라벨링** — 표시되는 모든 지표를 *후보 검색 커버리지/랭크*로 명시하고
   (시각/사용자 의도 기반 식별 정확도가 아님), Kakao Local 데이터가 준비될 때까지
   한국(Korea) 데이터는 보류(hold out).

오늘 총 6개 커밋이 반영되었습니다 (13:24 → 15:46).

## 4. MVP 채점 정책

채점 정책을 평가기·API·README·UI 전반에 걸쳐 명확하고 일관되게 정리했습니다.

- Kakao Local 정규(canonical)/후보 데이터가 채워질 때까지 **한국 데이터는 보류**
  (상태 플래그: `excluded_korea_pending_kakao`).
- **한국 외 행(row)** 은 **MapKit** 기준으로 평가.
- MVP 매칭은 **같은 provider 내 정확 문자열 일치**(정규 GT 이름 vs. 후보 이름) 사용.
- `provider_place_id` 는 nullable/선택 항목이며 MVP 채점에 **필수 아님**.
- 정규화(normalized)/문자열 완화 매칭은 **폴백/근거(evidence) 뷰로만 유지** —
  기본 provider-aware 정확 이름 정책을 대체하지 않음.
- 이 수치들은 **후보 커버리지/랭크 지표**이며, 시각/사용자 의도 기반 식별 정확도가 아님.

## 5. 구현 내용

### `tools/match_score.py` — provider-aware 지표 엔진
- 외부 의존성 없이 표준 라이브러리만 사용.
- CLI: `python3 tools/match_score.py --json`.
- 데이터셋 필터링 (`all`, `linkedspaces`, `union-city`, `vancouver`).
- exact/raw 모드 및 normalized 진단 모드.
- 현재 비공개 reconciled CSV 기반의 레거시 MapKit `app_poi_rank` 지원.
- 선택적 provider 후보 JSONL 로딩.
- MapKit TSV 미리보기 데이터(`ls_nearby_results.tsv`)를 생성 JSONL로 변환.
- 한국 보류 상태 플래그 출력.

### `server.py` — 라이브 매치레이트 엔드포인트
- `GET /api/matchrate?dataset=all&mode=exact`
- `GET /api/matchrate?dataset=linkedspaces&mode=exact`
- `GET /api/matchrate?dataset=all&mode=normalized`
- 목(mock) 수치를 반환하지 않고 `match_score.evaluate()` 를 직접 호출.
- 로컬 매치레이트 계산을 위해 비공개 데이터 경로를 읽도록 구성.

### `mvp-eval-ui.html` + `mvp-eval-ui.js` — UI 정리
- 평가 카드가 정적 목 수치 대신 `/api/matchrate` 를 fetch.
- **가짜/목 알고리즘 행 제거** (`ocr-match`, `nearest-only`, `fastvlm-name`,
  하드코딩된 22% / 15% 점수).
- 실행(run) 버튼이 더 이상 가짜 실행을 만들지 않음 — 제출 하네스(harness)가 아직
  구현되지 않았음을 명확히 표시.
- 실제 알고리즘 예측이 존재하기 전까지 선택 정확도(selection-accuracy) 차트는 비움.
- 엄격한 `script-src self` CSP가 UI를 막지 않도록 JavaScript를 HTML에서 분리해
  same-origin `mvp-eval-ui.js` 로 이동.
- 정적 개요(overview) 카운트/막대를 `/api/overview` 기반 값으로 교체.
- MapKit이 시각/사용자 의도 기반 POI 식별 정확도로 오해되지 않도록, 케이스 라벨을
  식별 실패 표현에서 후보 랭크 표현(`rank1`, `rank>1`, `검색실패`)으로 변경.
- 현재 채점이 한국을 제외하고 한국 외 행은 MapKit을 사용함을 UI 라벨에 명시.

### `tools/validate_upload_package.py` — 업로드 검증
- 인제스트(ingest) 전에 사용자가 채운 데이터셋 ZIP 패키지를 검증.
- 빈 템플릿 ZIP은 manifest 데이터 행이 없어 검증에 실패하는 것이 *정상* —
  사용자는 업로드 전에 `photos/` 와 `manifest.csv` 를 채워야 함.

### 생성된 로컬 산출물 (미공개)
- `generated/mapkit_candidates.jsonl` — `ls_nearby_results.tsv` 에서 변환한 537행.

## 7. 저장소가 아직 지저분해 보이는 이유

이 저장소는 새로 만든 것이 아니라 활성 프로토타입에서 분리한 것입니다. 첫 번째
우선순위는 *안전한 공개*(비공개/원본/생성 산출물 제거)였고, 그다음이 *실제* MVP
지표 연결이었습니다. 구조 정리는 의도적으로 점진적으로 진행합니다.

지저분함의 알려진 원인:

- `mvp-eval-ui.html` 은 여전히 단일 파일 프로토타입 (HTML + CSS + JS 혼재,
  다만 JS는 이제 `mvp-eval-ui.js` 로 분리됨).
- `server.py` 는 로컬 대시보드 서버로, 공개 저장소 밖의 로컬 데이터 파일이 존재한다고
  가정.
- 일부 오래된 스펙/리뷰/계획 문서는 `/api/matchrate` 와 `match_score.py` 이전에
  작성되어 상태 표현이 어긋나 있음.
- 생성/비공개 데이터가 제외되어 일부 명령은 비공개 작업공간에서만 동작.

정리 방침:

- 명시적 승인 없이 공개 Git 히스토리를 **재작성하지 않음**; 푸시된 커밋 유지.
- 정책·문서·보고서용 작은 forward 커밋 추가.
- `report/` 아래에 날짜별 작업 로그 추가.
- MVP 동작이 안정될 때까지 구조적 UI/서버 리팩터링은 보류.
